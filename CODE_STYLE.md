# Code Style

Conventions and patterns for the terraform monorepo. Enforced by TFLint, pre-commit hooks, and CI.

## Naming Conventions

### Terraform Identifiers

All Terraform identifiers use `snake_case`, enforced by TFLint `terraform_naming_convention`:

```hcl
# Variables, locals, outputs, resources, modules, data sources
variable "network_gateway" { ... }
locals { container_sizing = { ... } }
output "host_inventory" { ... }
resource "proxmox_virtual_environment_lxc" "this" { ... }
module "onepassword_secrets" { ... }
data "onepassword_vault" "this" { ... }
```

### File Naming

- Terraform files: `snake_case.tf` вЂ” `main.tf`, `variables.tf`, `outputs.tf`, `versions.tf`, `locals.tf`
- Templates: `kebab-case.tftpl` вЂ” `docker-compose.yml.tftpl`, `cloud-init.yaml.tftpl`
- Scripts: `kebab-case.sh` or `kebab-case.go`
- Directories: `kebab-case` with numeric prefix for workspaces вЂ” `100-pve`, `104-grafana`

### Resource Naming

- Single-instance resources use `"this"` as the name: `resource "x" "this" { ... }`
- Multi-instance resources use descriptive names: `resource "x" "grafana" { ... }`
- Module names match their purpose: `module "lxc"`, `module "config_renderer"`

## File Organization

### Standard Terraform File Layout

Every workspace and module follows this structure:

| File | Purpose |
| ---- | ------- |
| `main.tf` | Providers, resources, module calls |
| `variables.tf` | Input variables with descriptions and validation |
| `outputs.tf` | Output values with descriptions |
| `versions.tf` | `terraform {}` block: `required_version`, `backend`, `required_providers` |
| `locals.tf` | Local values (when non-trivial) |
| `data.tf` | Data sources (when present) |
| `BUILD.bazel` | Bazel governance |
| `OWNERS` | Code ownership |
| `README.md` | Auto-generated by terraform-docs (optional) |

## Variable Conventions

### Input Variables

All variables require descriptions (enforced by TFLint `terraform_documented_variables`):

```hcl
variable "node_name" {
  description = "Proxmox node name to deploy containers"
  type        = string
  default     = "pve3"

  validation {
    condition     = can(regex("^pve[0-9]+$", var.node_name))
    error_message = "Node name must match pattern 'pveN' (e.g., pve1, pve3)."
  }
}
```

Rules:
- Always set `description`.
- Always set `type`.
- Use `validation {}` blocks for constraints (regex patterns, ranges, non-empty checks).
- Mark secrets with `sensitive = true`.
- Prefer explicit `default` values where safe; omit for required inputs.

### Output Values

```hcl
output "host_inventory" {
  description = "Map of all provisioned hosts with metadata"
  value       = module.hosts.hosts
}
```

Rules:
- Always set `description`.
- Mark sensitive outputs with `sensitive = true`.
- Use `try()` wrappers for optional values.

### Locals

Group related locals with comment headers:

```hcl
locals {
  # Container sizing
  container_sizing = { ... }

  # VM definitions
  vm_definitions = { ... }
}
```

## Module Patterns

### Module Sources

Always use relative paths from workspaces:

```hcl
module "lxc" {
  source = "../modules/proxmox/lxc"
  ...
}
```

### Module Interface Contract

- Keep `variables.tf` and `outputs.tf` explicit with descriptions.
- Evolve by additive variables before breaking changes.
- Keep output key names stable вЂ” many templates consume them.

### for_each Pattern

Prefer `for_each` over `count` for named resources:

```hcl
module "lxc" {
  source   = "../modules/proxmox/lxc"
  for_each = local.containers

  vmid     = each.value.vmid
  hostname = each.value.hostname
  ...
}
```

### Host References

All IPs flow from the single source of truth (`envs/prod/hosts.tf`):

```hcl
# Correct
ip_address = module.hosts.hosts[each.key].ip

# Wrong вЂ” hardcoded IP
ip_address = "192.168.50.104"
```

## Template Conventions

### Template Files (`.tftpl`)

- Live in `{workspace}/templates/` directories.
- Rendered by `config-renderer` module via `templatefile()`.
- Output to `configs/` directories (read-only, regenerated by `terraform apply`).

### Template Variables

Templates receive a `template_vars` map containing:

```hcl
template_vars = {
  hosts                  = module.hosts.hosts        # Full host inventory
  grafana_admin_password = module.secrets.secrets.grafana.admin_password
  # ... service-specific vars
}
```

### Template Access Pattern

```
# In .tftpl files
${hosts.elk.ip}:${hosts.elk.ports.elasticsearch}
${grafana_admin_password}
```

## Secret Handling

- All secrets retrieved via `modules/shared/onepassword-secrets/`.
- Access pattern: `module.onepassword_secrets.secrets["key_name"]`.
- Wrap with `try()` for fallback: `try(module.secrets.secrets["key"], "")`.
- Use `sensitive = true` on variables and outputs containing secrets.
- Never commit `.tfvars`, `.env`, `.tfstate`, or API keys.
- Never hardcode secrets in `.tf` or `.tftpl` files вЂ” use `op://` references.

## Provider Configuration

### Provider Blocks

Providers declared in the workspace `main.tf`, version constraints in `versions.tf`:

```hcl
# main.tf
provider "proxmox" {
  endpoint  = var.proxmox_endpoint
  api_token = local.effective_proxmox_api_token
  insecure  = var.proxmox_insecure
}

# versions.tf
terraform {
  required_version = ">= 1.7, < 2.0"
  backend "local" {}
  required_providers {
    proxmox = {
      source  = "bpg/proxmox"
      version = "~> 0.94"
    }
  }
}
```

### Version Constraints

- Use `~>` (pessimistic) for provider versions.
- Use range constraints for Terraform itself: `>= 1.7, < 2.0`.

## Validation

### TF 1.5+ Check Blocks

Use `check {}` blocks for runtime validation in `checks.tf`:

```hcl
check "vmid_range" {
  assert {
    condition     = ...
    error_message = "VMID out of managed range."
  }
}
```

### Variable Validation

Prefer `validation {}` blocks over runtime errors:

```hcl
variable "proxmox_endpoint" {
  ...
  validation {
    condition     = can(regex("^https://", var.proxmox_endpoint))
    error_message = "Proxmox endpoint must use HTTPS."
  }
}
```

## Formatting and Linting

### Terraform Format

`terraform fmt` enforced via pre-commit and CI. Run locally:

```bash
make fmt           # Format all .tf files
make lint-tf       # Check formatting (no write)
```

### TFLint

Config: `.tflint.hcl` вЂ” terraform plugin v0.10.0 with `recommended` preset.

Enforced rules:
- `terraform_naming_convention` вЂ” `snake_case` for all identifiers
- `terraform_documented_variables` вЂ” descriptions required
- `terraform_documented_outputs` вЂ” descriptions required
- `terraform_standard_module_structure` вЂ” standard file layout

### Pre-commit Hooks

6 repos, run via `pre-commit run --all-files`:

- `pre-commit-hooks`: trailing-whitespace, end-of-file-fixer, check-yaml, check-json, check-merge-conflict, detect-private-key
- `pre-commit-terraform`: terraform_fmt, terraform_validate, terraform_tflint, terraform_docs
- `yamllint`: YAML linting (`.yamllint.yml` config)
- `shellcheck`: Shell script linting (`--severity=warning`)
- `detect-secrets`: Secret detection (`.secrets.baseline`)
- `actionlint`: GitHub Actions workflow linting

## Testing Patterns

Tests use Terraform's native test framework (`terraform test`):

```bash
make test              # All tests
make test-unit         # Module unit tests
make test-integration  # Integration tests
make test-workspace    # Workspace validation
```

Test directories mirror the module/workspace structure under `tests/`.

## CI/CD Conventions

### GitHub Actions

- All actions SHA-pinned with `# vN` version comment suffix:

  ```yaml
  uses: hashicorp/setup-terraform@5e8dbf3c6d9deaf4193ca7a8fb23f2ac83bb6c85 # v4
  ```

- Never use mutable tags (`@v4`).
- Reusable workflows prefixed with `_` (e.g., `_ci-node.yml`).

### Deployment

- `make apply` is disabled locally вЂ” all applies via CI/CD.
- PR triggers `terraform plan`; merge to master triggers `terraform apply`.
- CI concurrency groups prevent parallel applies per workspace.

## Commit Conventions

- **Format**: `type(scope): imperative summary` (в‰¤72 chars, lowercase)
- **Types**: `feat`, `fix`, `docs`, `refactor`, `test`, `ci`, `chore`, `perf`, `build`, `revert`
- **Branch naming**: `type/description` (e.g., `feat/add-metrics-export`)
- **PR size**: ~200 LOC max. Squash merge only.
- **Merge policy**: Squash merge only. Merge commits disabled.

## Anti-Patterns

- **NO** hand-editing `configs/` or `tf-configs/` вЂ” always regenerate via `terraform apply`
- **NO** local `make apply` вЂ” deploy via CI/CD only
- **NO** committing `.tfvars`, `.env`, `.tfstate`, or secrets
- **NO** hardcoded IPs вЂ” use `hosts.tf` variables via `module.hosts`
- **NO** mutable GitHub Action tags (`@v4`) вЂ” SHA-pin all actions
- **NO** Docker socket mounting in any workspace
- **NO** manual LXC/VM resource changes in Proxmox UI вЂ” Terraform-managed
- **NO** type error suppression (`as any`, `@ts-ignore`) in any scripting
- **NO** inline cloud-init вЂ” use external templates
- **NO** direct resource duplication вЂ” use modules
